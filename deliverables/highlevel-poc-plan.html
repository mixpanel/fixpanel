The gap we're closing

For a given parent account (e.g., Boutique Del Fitness), HighLevel needs an aggregated view of all events across all child accounts combined. Pendo cannot produce this view. That's the canonical group analytics use case R2 on your scorecard and gap 1 in your onboarding doc. The model below makes it native.

Entity → Mixpanel primitive

Your four HighLevel entities map directly to Mixpanel's data model: two group keys for account hierarchy, and the user collection for the people performing actions.

Three calls worth confirming

1

Two group keys, not three. Your location_id is the same row as a sub-account not a third entity. The relation from Parent Account to Sub Account is one-to-many.

2

No parent → child denormalization needed. Child events carry both parent_account_id and sub_account_id as group IDs. "All events for Boutique Del Fitness" is one group filter, no cross-join or array maintenance.

3

Contacts are users, not a group. They're the actors firing portal events. Promoting them to a group key misses out on Mixpanel's identity management, which is a key part of the POC.

Event payload contract

Every event that can be attributed to a Parent Account or Sub Account ships the IDs needed for rollup at fire time.

Rollup in action

For parent_account = Boutique Del Fitness: sum of all events with that group ID = parent-level events + 47 sub-accounts + portal contacts. One group filter. No SQL.

Identity

Two populations, one project. The user_type property keeps them sortable.

Goal: 3-value enum

agency_user sub_account_operator portal_contact

The ideal state if Pendo's payload carries the agency-vs-operator signal.

Fallback: 2-value enum

platform_user portal_contact

Used if the current Pendo stream doesn't distinguish agency staff from sub-account operators.

distinct_id strategy

Trust Pendo's existing namespace. Assuming Pendo was instrumented to create anonymous $device_id values and store user_id once contacts log in, preserving that namespace keeps the anonymous → identified merge intact post-migration.

Week-1 audit confirms: (a) no collision between contact and platform user IDs, and (b) that the device-to-user merge is actually happening in their current data. Prefix at ETL if either check fails.
